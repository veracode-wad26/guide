Supply Chain Security Overview

What we're covering in the presentation

Why this matters

62% of applications contain vulnerabilities from third-party libraries. Those vulnerabilities take an average of 358 days to fix. In 2025, Sonatype identified over 454,600 new malicious packages across npm, PyPI, Maven, NuGet, and Hugging Face.

What we'll discuss

1
Supply chain definition
Third-party SaaS, libraries, IDE extensions, CI/CD pipelines, transitive dependencies, base images, and AI tools — everything external that touches your software.
2
Real-world attacks
SolarWinds (18,000 organizations compromised), Bybit ($1.5B theft via wallet software), and recent malicious package campaigns.
3
Common attack vectors
Typosquatting, dependency confusion, malicious injection, repo/package takeover, CI/CD poisoning, and code obfuscation techniques.
4
Malware analysis deep dive
How attackers layer obfuscation, steganography, and encoding across 12+ stages to hide malicious payloads in packages.
5
AI agent risks
Over 36% of AI skills contain vulnerabilities. Skills can access files, shells, and credentials — treat them like untrusted third-party code.
6
Prevention strategies
Lock files, minimal release age policies, sandboxing, manual code review, and enterprise filtering tools.

After the presentation, we'll move into hands-on exercises where you'll fix real vulnerabilities and see defense mechanisms in action.

The Exploit, Live Demo

SCA vulnerability → sandbox escape → package firewall block

Part 1 — SCA Scanner finds the vulnerability

The demo app ships with vm2 3.9.5 — a JavaScript sandbox library with 27 Critical CVEs. The SCA scan surfaces them instantly.

1
SCA scan output
27 Critical CVEs in vm2 3.9.5 alone — multiple sandbox escape vulnerabilities, several scoring CVSS 10.0.
2
vm2 — a broken sandbox
vm2 promises to run untrusted JavaScript safely. CVE-2023-29199 breaks that promise — an attacker can escape the sandbox entirely and run arbitrary code on the host.
3
Live RCE exploit
One curl command breaks out of the sandbox, reads credentials off disk, and proves full system access on the host.

Part 2 — Package Firewall blocks a malicious install

A package firewall blocks malicious packages — typosquats, supply chain attacks, intentionally backdoored packages. It does not fix CVEs; that's what the SCA scanner is for.

4
Attempt to install a malicious package
npm install @veracode/firewall-demo@1.0.0 — this package is registered as a known malicious demo package. Your firewall should block it immediately.
5
The block
The install is blocked. Your firewall caught the package before it touched your environment. No code ran, nothing was installed.
⚠ Check with your security team first. The Veracode demo packages are safe and contain no malicious code, but adding them may trigger your firewall. Let your security or platform team know so they expect it.

Step 2, Clone and open locally

After forking, clone your fork to your machine and open in your preferred editor.

git clone https://github.com/YOUR-USERNAME/verademo-javascript-api.git && cd verademo-javascript-api && git checkout participant-start
ℹ The participant-start branch is the default, the intentionally vulnerable version. All your changes will be on this branch.

Step 3, Scan for vulnerabilities

Run OSV Scanner locally to see all Critical and High findings in your repo.

# Install osv-scanner (once)
brew install osv-scanner                   # macOS
winget install Google.OSVScanner            # Windows

# Run the scan
osv-scanner scan -r .
# Example output

CRITICAL · CVE-2021-23337 · lodash 4.17.20
Command injection via _.template(). Fix: bump to 4.17.21

HIGH    · CVE-2023-45857 · axios 0.21.1
CSRF via forged headers. Fix: bump to 1.6.0

Step 4, Fix without breaking

Edit only the dependency manifest. The app must still pass its test suite after your changes.

Your manifest file
package.json
# Verify locally first
osv-scanner scan -r .

# Commit and push to trigger the scan + score
git add . && git commit -m "fix: remediate critical CVEs"
git push

Step 5, Push and watch the scan

Your push triggers the GitHub Actions workflow. An OSS scanner runs against your manifest automatically, no credentials needed. Check the Actions tab in your fork.

# What runs in Actions (all ecosystems)

osv-scanner scan -r -f json .

# Scores your PR against the baseline and updates the leaderboard
Still failing ❌
Check the job log, a dep may still be pinned to a vulnerable version, or a transitive is pulling in something old. Go back to step 4.
All clear ✅
Zero Critical/High findings. Raise your hand, you're on the leaderboard.
⚠ If you see findings you didn't expect, check for transitive dependencies, a package you kept may pull in a vulnerable version indirectly. Look at the full audit output, not just the top-level deps.

Q&A + Best Practices

Open discussion, Boy Baukema leads

No slides. Bring your questions from the exercise. We'll work through real scenarios.

Discussion topics we'll cover

①
Where does SCA stop and the Package Firewall begin?
SCA tells you what's vulnerable. The Package Firewall stops malicious packages before they're ever installed. They're complementary, not competing.
②
Blocking vs. alerting, what's right for your pipeline?
Hard blocks in CI slow dev teams. Alerting without action creates noise. How do you find the right policy for your risk tolerance?
③
Breaking changes when remediating, how do you manage this?
Major version bumps break APIs. Strategies: test coverage first, pin transitives, use Veracode Fix for automated suggestions.
④
Where to go from here
Developer-first security: IDE plugins, pre-commit hooks, SCA in PR comments. Security that gets out of the developer's way.

Key concepts from today

!
SCA is not optional
Your app is 80%+ open-source code by dependency count. You own the risk from every package you import, whether you wrote it or not.
⊕
Shift left, not shift all responsibility left
Put SCA in the IDE and the PR, but back it up with pipeline policy. Developers need guardrails, not gates.
✓
Remediation is the end goal, not detection
A long vuln backlog isn't a security program. Prioritise by reachability, fix by version bump where possible, track progress over time.
